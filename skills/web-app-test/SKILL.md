# 网页应用自动化测试 Skill

## 依赖

本 skill 依赖 **`playwright-cli` skill**（不随本仓库分发）。它由 npm 包 [`@playwright/cli`](https://www.npmjs.com/package/@playwright/cli) 提供：装包即同时得到 `playwright-cli` 命令行工具**和**完整的 skill 文档（`SKILL.md` + `references/*.md`）。

### 安装

```bash
# 1. 全局安装 playwright-cli（提供命令行工具 + skill 文档）
npm install -g @playwright/cli@latest

# 2. 安装浏览器驱动（首次使用前执行一次）
playwright-cli install

# 3. 验证安装
playwright-cli --version
```

> 不想全局安装时，可用本地方式：`npx --no-install playwright-cli --version` 验证可用后，把后续所有命令的 `playwright-cli` 换成 `npx playwright-cli`。

### skill 文档位置

安装后，`playwright-cli` 的 skill 文档位于全局 node_modules 内：

```
<global node_modules>/@playwright/cli/skills/playwright-cli/
├── SKILL.md
└── references/*.md
```

用 `npm root -g` 查出 `<global node_modules>` 的实际路径。本 skill 会引用其中：
- AI 动态接入浏览器的操作命令（默认方式）见 `playwright-cli` 的 `SKILL.md`
- 沉淀核心流程为自动化脚本时，编写、调试、生成见 `playwright-cli` 的 `references/test-generation.md`、`spec-driven-testing.md`、`playwright-tests.md`

执行浏览器测试前，确认 `playwright-cli` 命令可用、其 skill 已加载。

## 测试分类

测试分为两大类，**两类的脚本设计策略相反**：

### 第一类：程序测试脚本（必须预先设计，覆盖主要功能）

脚本直接调用后端模块和 API，验证数据层、配置层、核心逻辑、API 契约、前端构建等。不涉及浏览器。

**这类脚本必须预先设计好**：创建/维护规格书时，就要为 L0–L3、L5 的每条用例写出对应的可执行脚本或可直接运行的命令，并**覆盖产品的主要功能**（核心数据、关键配置、核心生成/处理流程、主要 API 端点、错误处理、生产构建等）。规格书里描述了却没有对应脚本/命令的，视为未完成。这类脚本是回归的底座，不依赖现场判断，应稳定存在、可随时无人值守复跑。

### 第二类：浏览器模拟用户操作（Playwright 脚本不预先设计，动态沉淀）

用 Playwright 驱动真实浏览器，模拟用户在前端界面上的完整操作流程。**连接真实后端，不做 mock。** 前后端集成行为与 API 契约同时验证。涉及 LLM 调用的用例（生成/精炼）超时设 180s+；涉及 LLM 输出的断言用「至少 N 个」而非精确数。

**这类的 Playwright 脚本不预先静态设计**：默认由 AI 动态接入浏览器现场操作，AI 把某个功能测完之后，再现场判断「后续是否需要回归」，需要的才沉淀为 Playwright spec（见下文「浏览器测试的执行模式」）。浏览器界面演进快，预写脚本容易过时，动态接入更灵活。

## 核心原则

1. **真实集成，不做 mock**：浏览器测试连接真实运行的后端，验证前后端真实集成行为。不拦截、不伪造 API 响应。
2. **两种验证手段并用**：
   - **API 返回值 + 后端日志**：每次操作后检查 API 实际响应内容、HTTP 状态码，并读取后端处理日志确认执行路径正确。
   - **截图 + 视觉判断**：每个关键步骤后截图，利用视觉能力（多模态 LLM）观察前端当前状态是否符合预期。
3. **最大化测试覆盖**：每轮尽量跑完所有测试用例，不因单个失败提前停止。只有当前序失败**真正阻断**后续测试（如后端崩溃导致所有 API 用例无法执行、登录失败导致所有需认证的用例无法进行）时才跳过后续。记录每个跳过的用例及跳过原因，修复前序问题后补跑。**不要因为「有失败」就停下来——多收集失败信息，一次报告，比多轮测试效率高。**
4. **分层 fail-fast 仅限自动化运行器**：`run_regression.py` 等 CI 脚本的 fail-fast 是为了节省 CI 资源，AI 执行时不受此约束，应遵循原则 3。
5. **非交互**：所有脚本禁止 `input()` 或交互式提示，确保可自动运行。
6. **产物可追溯**：截图、日志、API 响应保存到产物目录，便于事后分析。
7. **每轮测试必须交付 Markdown 测试报告**：每次测试结束，都要产出一份 Markdown 格式的测试报告交付给用户，不能只在对话里口头总结。报告内容、命名与存放位置见下文「测试结束：交付 Markdown 测试报告」一节。
8. **脚本设计策略分两类**：程序测试脚本（后端/API/构建，L0–L3、L5）**必须预先设计并覆盖主要功能**，是回归底座；只有浏览器 E2E 的 **Playwright 脚本不预先设计**——由 AI 动态接入浏览器把功能测完后，现场判断「这条测试后续是否需要回归、是否值得沉淀成 Playwright 脚本」再生成。见下文「测试分类」与「浏览器测试的执行模式」。

## 项目专属内容：每次测试前先审查

本 skill 严格区分两类内容，**分别存放在不同位置**：

| 类别 | 内容 | 存放位置 |
|------|------|----------|
| **通用 skill 本体**（不要项目化） | `SKILL.md`（本文件）、`templates/` | skill 目录内（随 skill 分发） |
| **项目专属实例**（每次测试前审查更新） | `TEST_SPECIFICATION.md`、`L4_OPERATION_GUIDE.md` 等分支文档、`scripts/` | **被测项目根目录下的 `web-app-test/` 文件夹**（随项目走，不进 skill 目录） |

> **为什么分开放？** skill 本体是可跨项目复用的通用资产，通过 skills 分发渠道安装，对所有项目只有一份；而 `TEST_SPECIFICATION.md` / `scripts/` 绑定具体项目的功能、API、选择器、端口，属于**被测项目**的一部分，应该和项目代码一起进项目自己的 git 仓库。把两者混在 skill 目录里，会导致 skill 无法干净分发、且项目专属内容游离在项目仓库之外。

### 项目专属内容的约定位置

```
<被测项目根>/web-app-test/
├── TEST_SPECIFICATION.md      # 测试用例规格书（主文档，由 templates 落地生成）
├── L4_OPERATION_GUIDE.md      # 分支文档示例（按需）
├── <其他分支文档>.md
└── scripts/                   # 可执行测试脚本（全部在规格书附录登记并链接）
```

执行测试时，先在被测项目根目录查找 `web-app-test/`。下文凡提到 `TEST_SPECIFICATION.md`、`scripts/`、分支文档，均指此目录下的文件。

**执行测试前，必须先对 `web-app-test/TEST_SPECIFICATION.md` 和 `web-app-test/scripts/` 做一次「判断 → 创建或维护」：**

> **规格书必须声明所属产品，并管理所有脚本。** `TEST_SPECIFICATION.md` 开头第一段必须写清「本文档是 `<产品名>` 的测试用例规格书」，让任何人一眼知道这套测试针对哪个产品。同时，`scripts/` 下的**每个可执行脚本都必须被规格书管理并链接**——在规格书的「脚本清单」附录里登记脚本路径、对应用例、复跑命令。规格书是脚本的唯一索引，不允许出现「孤儿脚本」（存在于 `scripts/` 但规格书里查不到）。新增/删除脚本时同步更新该附录。
>
> **规格书不是设计文档。** `TEST_SPECIFICATION.md` 中的每条用例必须挂有实际可执行资源，否则 agent 无法执行测试。可执行资源的形式取决于用例类型：
>
> | 用例类型 | 可执行资源形式 |
> |----------|---------------|
> | 后端模块/逻辑测试（L0–L2） | `scripts/` 中的 Python 脚本，或可直接运行的 `python -c` / pytest 命令 |
> | 前端构建/静态检查（L3） | 规格书中写明可直接执行的 `npm run ...` / `npx ...` 命令 |
> | 浏览器 E2E（L4） | 默认由 AI 动态接入浏览器执行（依据规格书中的用例目标 + 用户指南现场操作）；凡后续有必要回归的功能，AI 测完后现场判断是否沉淀为 `scripts/` 中的 Playwright spec 函数（沉淀决策见「浏览器测试的执行模式」） |
> | 真实 API 集成（L5） | 规格书中写明可直接执行的 `curl` / HTTP 请求命令，或 `scripts/` 中的脚本 |
>
> **创建或维护规格书时，每条用例必须同步确认其可执行资源存在且有效。** 脚本的设计策略按类型分两种：
> - **后端/API/构建类用例（L0–L3、L5）→ 必须预先设计**：规格书创建时就要写好对应脚本或可直接运行的命令，并覆盖产品主要功能。规格书里描述了但没有对应脚本/命令的，视为未完成。
> - **浏览器 E2E 的 Playwright spec（L4）→ 不预先设计，动态沉淀**：默认由 AI 动态接入浏览器执行，规格书只需写清用例目标与验收标准即可——不必预先为每条都写 spec。spec 由 AI 动态测完该功能后，现场判断「后续是否有必要回归」，有必要的才沉淀。

1. **判断这两部分是否已存在且针对当前项目**：
   - **不存在 / 是从别处拷来的他项目内容** → 这是一个新接入的项目。此时执行「**新项目落地**」流程（见下）。
   - **已存在且针对当前项目** → 进入下一步维护。

   #### 新项目落地流程

   当被测项目根目录下**没有 `web-app-test/`**，或其中内容针对的是别的项目时：

   1. 在**被测项目根目录**创建 `web-app-test/` 文件夹（及其下的 `scripts/`）。
   2. 阅读当前项目的前端源码、后端源码、API 文档、用户指南，摸清功能点、API 契约、选择器、目录与端口。
   3. 以 skill 目录内 `templates/` 下的模板为骨架，**拷贝到 `web-app-test/` 并按下表落地**：

      | 模板（skill 目录内） | 落地为（被测项目根） |
      |----------------------|----------------------|
      | `templates/TEST_SPECIFICATION.template.md` | `web-app-test/TEST_SPECIFICATION.md` |
      | `templates/run_regression.template.py` | `web-app-test/scripts/run_regression.py` |
      | `templates/playwright.config.template.js` | `web-app-test/scripts/playwright.config.js` |
      | `templates/standard-flow.spec.template.js` | `web-app-test/scripts/<flow>.spec.js` |

   4. 按项目实际替换模板中的占位符 `<...>`，删除用不到的层级/用例，覆盖该项目真实的功能点、API 契约、选择器契约、目录与端口。模板用法、占位符约定见 **[templates/README.md](templates/README.md)**。
   5. 落地后，`web-app-test/` 应和被测项目代码一起纳入该项目的版本控制。

2. **维护已有内容**：项目代码持续演进，测试用例和脚本可能因此过时（选择器失效、流程变化、新功能未覆盖、API 契约变更）。执行测试前应**主动判断测试是否仍然反映当前系统行为**，对照源码与用户指南，按需调整规格书与脚本，再开始跑测试。

判断信息源优先级、Feature vs Bug 的甄别方法，见下文「测试维护原则」一节。**不要在测试预期过时的情况下盲目跑测试或盲目改测试使其通过——先判断根因。**

## 测试环境准备（AI 负责，不依赖用户手动操作）

测试所需的服务（前端 dev server、后端 API server 等）由 **AI 自行启动**，不要要求用户手动操作。具体启动命令、端口、健康检查方式记录在项目专属的 `TEST_SPECIFICATION.md` 附录中（参考「附录 C：浏览器测试环境配置」一节）。

### 执行流程

1. **检查服务状态**：先用健康检查命令确认前后端是否已在运行（如 `curl localhost:8085/api/health`、`curl localhost:5173`）。已在运行则直接使用，不重复启动。
2. **按需后台启动**：若服务未运行，用后台命令启动（`nohup ... &` 或 `start /B ...`），等待健康检查通过再继续。
3. **启动失败则报告**：若服务启动失败（超时、端口冲突、依赖缺失），在测试报告中说明原因，跳过依赖该服务的测试层，继续跑不依赖该服务的层（如 L0/L2 不依赖后端）。
4. **测试完成后保留服务**：不主动关闭用户可能仍在使用的服务；若是 AI 本轮启动的且用户无需继续使用，可关闭——不确定时保留。

### 规格书中的记录要求

`TEST_SPECIFICATION.md` 的「附录 C」或等价章节必须包含：

| 项 | 内容 |
|----|------|
| 前端启动命令 | 如 `cd behavior-tree-editor && npm run dev` |
| 后端启动命令 | 如 `cd bt_server && python server.py` |
| 健康检查命令 | 如 `curl -s localhost:8085/api/health` |
| 端口 | 前端固定端口、后端固定端口 |
| 环境前提 | conda 环境名、node 版本要求等 |
| 端口冲突处理 | 如何检查并释放端口 |

新接入项目时，创建 `TEST_SPECIFICATION.md` 必须同步填写此表。

## 浏览器测试的执行模式

**默认 AI 动态接入浏览器，不要事先为每个测试用例都写好自动化脚本。** 浏览器界面演进快，预写脚本容易过时（选择器失效、流程变化、新功能未覆盖），逐个维护成本高。AI 直接驱动浏览器更灵活，能即时适应界面变化、按用户视角判断真实表现。

### 默认方式：AI 动态接入浏览器

对绝大多数测试用例，AI 直接驱动浏览器，执行"观察→决策→操作→验证"循环：

1. 读取当前前端源码和用户指南，理解最新的界面结构
2. 截图观察当前页面状态
3. 根据测试目标和页面实际内容，决定下一步操作（不依赖预设选择器）
4. 通过 Playwright CLI 执行操作（具体命令参考 `playwright-cli` skill）
5. **积极截图，按用户视角判断**：每个关键步骤后截图，用视觉能力观察「如果我是用户，这个产品现在表现正常吗」
6. **前后端双向验证**：除了前端视觉检查，同时检查后端状态与日志（API 实际响应、HTTP 状态码、后端处理日志的执行路径），确认行为符合预期，而不只是界面看起来对
7. 循环直到该用例验证完成

### 沉淀方式：动态测完后判断是否沉淀为自动化脚本

**沉淀不是提前静态设计的，而是 AI 动态接入浏览器把某个功能测完之后的现场决策。** 每测完一个功能用例，AI 都要追问一句：**「这条测试后续有没有必要回归？」** 凡是后续有必要回归的功能，就把它沉淀为可复跑的 Playwright 自动化脚本；一次性、探索性、或几乎不会再变也不会再测的用例，不必沉淀。

判断「是否值得沉淀」的依据：

- **后续是否需要回归**（核心判据）：该功能是产品的稳定能力、关键路径，或改动频繁、容易被其他改动破坏 → 需要回归 → 沉淀为脚本。
- **流程是否足够稳定可无人值守复跑**：步骤明确、断言可程序化、选择器相对稳定 → 适合写成 spec。
- 若流程相对固定但选择器/细节易小幅变动，可改为写一份**详细操作流程文档**（点哪里、填什么、看什么），AI 之后直接参考它动态操作，省去重新摸索界面。这类操作手册可独立成分支文档（如 `L4_OPERATION_GUIDE.md`）——见下文「分支文档」说明。

两种沉淀形式：

- **Playwright 脚本**：后续需回归、流程足够固定、值得无人值守批量复跑的，测完后立即在 `scripts/` 中生成对应 spec，之后可直接执行。
- **详细操作流程文档**：流程相对固定但细节可能小幅变动的，沉淀为操作步骤说明，AI 后续参考它动态操作。

对沉淀下来的脚本/流程，执行后 AI 仍要做二次判断：
1. **审查失败项**：脚本报"不通过"时，AI 读截图和 API 响应，判断是真实 bug 还是脚本/流程过时（选择器变了、流程调整了）。若是过时，AI 动态操作浏览器重新验证。
2. **抽查通过项**：对可疑的通过项（如截图显示异常但断言恰好通过），主动截图、点击、检查，确认无隐藏问题。
3. **补充覆盖**：对比用户指南，发现脚本/流程未覆盖的新功能或变更流程，动态生成操作进行补充测试。

### 执行流程

```
AI 动态接入浏览器（默认）
  ├── 已有脚本/操作手册的核心流程 → 直接执行/参考执行
  └── 其余用例 → 现场观察→决策→操作
  ↓
每步：截图（用户视角判断）+ 检查 API 响应/HTTP 状态码 + 读后端日志
  ↓
每测完一个功能 → 判断「后续是否需要回归」→ 需要则现场沉淀为 Playwright spec
  ↓
对失败/可疑项：判断是 bug 还是脚本过时 → 动态重测或确认 bug
  ↓
输出：① Markdown 测试报告（确认的 bug + 脚本/流程需更新项 + 新发现的问题）
      ② 本轮新沉淀/更新的 Playwright 脚本
```

## 浏览器测试的验证方法

判断"操作是否正确"不依赖 mock 的预设返回值，而是：

### 方法 A：检查 API 返回值和后端日志

```
操作：点击某个触发 API 调用的按钮
  → 等待对应 API 响应
  → 断言：response 状态码和关键字段符合预期
  → 读取后端日志最近几行，确认执行路径正确
```

适用于：有明确 API 交互的操作。

### 方法 B：截图 + 视觉判断

```
操作：触发某个 UI 变化
  → 截图保存到产物目录
  → 用视觉能力观察：界面状态是否符合预期？
```

适用于：画布渲染、主题切换、面板展开/折叠、布局变化等视觉状态验证。

### 实际测试中两种方法结合使用

一次完整的操作验证：
1. 执行用户操作
2. 等待 API 响应 → 检查返回数据结构（方法 A）
3. 等待 UI 更新 → 截图 → 视觉确认状态正确（方法 B）
4. 读后端日志 → 确认无异常（方法 A）

## 测试结束后：分类处理，不一律等待

测试执行完成后，按失败类型决定是自行处理还是等待用户：

- **类型 A（fixture 缺陷）/ 类型 B（Feature 变更）**：直接修复，修复后在报告里说明做了什么，不需要等用户确认。
- **类型 C（意外 Bug）**：在报告里列出，等用户确认修复方向再动手。

整理测试结果清单的格式如下：

### 测试结果清单格式

```
## 测试结果

### 通过 ✓
- T4-1 页面加载与初始状态
- T4-6 后端健康检查与配置
- ...

### 失败 ✗
- T4-X 用例名
  - 现象：...
  - 判断：Bug / 脚本过时 / 环境问题
  - 建议：...

### 跳过 / 未执行
- T4-X 用例名（原因：后端未启动 / 依赖前置用例失败 / ...）
```

### 后续步骤

- **类型 A/B（已自行修复）**：回归受影响的用例，确认修复有效且无新问题；补跑本轮因环境原因跳过的用例。
- **类型 C（Bug，用户确认后）**：按用户指定方向修复代码 → 回归受影响用例 → 补跑跳过项。

## 测试结束：交付 Markdown 测试报告

**每一轮测试结束都必须产出一份 Markdown 测试报告交付给用户**，不能只在对话里口头总结。报告是本轮测试的正式交付物。

- **存放位置**：产物目录下，建议 `test_results/report_<日期时间>.md`（沿用项目 `TEST_SPECIFICATION.md` 中约定的产物目录）。
- **交付时机**：整轮测试完整跑完（所有 📜 脚本用例已执行、🤖 动态用例已执行或明确标注跳过、已发现 bug 已修复并回归通过）后产出。中途中断不出最终报告，但应按 CLAUDE.md「测试进度可见性」要求输出进度快照。
- **生成后**：在对话中给出报告文件路径，并附简短摘要（通过/失败/跳过计数 + 关键结论）。

### 报告内容结构（建议）

```markdown
# 测试报告 <日期时间>

## 概要
- 范围：本轮覆盖的层级/用例范围
- 结果：通过 N / 失败 M / 跳过 K
- 关键结论：一句话总览

## 环境
- 前端/后端版本或 commit、端口、模型、conda 环境

## 测试结果明细
（沿用上文「测试结果清单格式」：通过 ✓ / 失败 ✗ / 跳过）

## 发现的问题
- Bug 列表（现象、判断类型 A/B/C、处理状态）

## 本轮已修复
- 类型 A/B 的自行修复说明 + 回归结果

## 沉淀的自动化资产
- 本轮新增/更新的 Playwright 脚本及其覆盖的功能、复跑命令

## 待办 / 需用户确认
- 类型 C bug 的修复方向待确认项
```

## 测试维护原则：代码变更时的测试同步

项目代码持续演进，测试用例和脚本可能因此过时。执行测试前，应主动判断测试是否仍然反映当前系统行为，并按需调整。

### 判断依据（信息源优先级）

1. **用户指南** — 最高可信度，代表产品当前设计意图
2. **API 参考文档** — API 契约的权威来源
3. **前端源码** — 实际实现，反映真实行为
4. **后端源码** — 同上
5. **测试脚本** — 可能滞后，需要被校验

当测试预期与用户指南/源码冲突时，**以用户指南和源码为准**，测试需要更新。

### 失败原因分类与处理方式

当测试失败时，不要盲目修复测试使其通过——先判断根因，按类型决定谁来处理：

#### 类型 A：测试脚本 fixture / 构造问题（AI 自行修复）

测试脚本内部逻辑有缺陷，与被测系统无关。被测系统行为正确，是脚本造成了误报失败。**AI 应直接修复脚本，无需询问用户。**

识别特征：
- 脚本里构造的 fixture 数据不完整（如自定义节点用进了树但未注入上下文）
- 测试用例的前置条件设置遗漏（如缺少必要的初始化步骤）
- 断言逻辑错误（如精确数断言了 LLM 输出、断言了内部实现细节而非外部行为）
- 选择器/路径写死了已变更的内部结构

修复原则：让 fixture 正确反映真实用法；不改变被测系统的行为；修复后回归确认通过。

#### 类型 B：Feature（预期变更，AI 自行更新测试）

用户指南已描述新行为，或源码有意重构了逻辑。旧测试预期已过时。**AI 更新测试的预期结果和操作步骤，无需询问用户。**

识别特征：用户指南描述的行为与测试预期不符；git commit message 明确说明了该变更；新源码实现有别于测试期望但本身逻辑完整。

#### 类型 C：Bug（意外回归，汇报用户）

用户指南未提及此变化，且旧行为是正确的。**保留测试预期，将 bug 汇报给用户后再修复代码。**

识别特征：用户指南仍描述旧行为；无相关 commit 说明变更；系统输出与文档承诺不符。

#### 判断流程

1. 先看失败信息——fixture 构造问题？断言逻辑问题？（→ 类型 A，直接修）
2. 读用户指南中对该功能的描述（→ 行为变更？类型 B；文档与现实不符？类型 C）
3. 查看相关 git commit message，确认是否为有意变更（→ 强化 B/C 判断）
4. 类型 A/B 自行修复；类型 C 汇报用户

## 分支文档：主文档之外的详细说明

测试说明不必全部挤进主文档。**允许在 `SKILL.md` / `TEST_SPECIFICATION.md` 主文档之外，另建分支文档承载更详细、更专门的说明。** 主文档负责「全局视角、原则、用例索引」，分支文档负责「某一类测试的逐步操作细节」。

### 何时拆分支文档

- 某一层 / 某一类用例的操作步骤细节多、篇幅大，写进主文档会冲淡全局视角
- 内容脆弱、随界面演进而频繁变动（如 L4 浏览器 UI 的逐步点击流程），单独维护更清晰
- 同一类操作有大量可复用的「点哪里、填什么、看什么」步骤，值得集中成一份操作手册供 AI 现场参考

### 约定

- 分支文档与 `TEST_SPECIFICATION.md` 一起放在被测项目根的 `web-app-test/` 目录下，命名能自解释其定位（如 `L4_OPERATION_GUIDE.md` 表示 L4 浏览器操作手册）
- **主文档必须引用分支文档**：在 `TEST_SPECIFICATION.md` 的对应章节或目录结构中留下指针，AI 才能发现并按需加载
- 分支文档同样遵循「项目专属内容每次测试前审查」的原则——界面/流程变了就同步更新
- 现有示例（BTGen 项目）：`web-app-test/L4_OPERATION_GUIDE.md` —— L4 浏览器动态接入的逐用例操作手册，覆盖导入/导出、撤销重做、清除重置、主题切换、设置面板、流式生成、空画布引导、变量字典等脆弱 UI 流程

## 目录结构

本 skill 涉及两处目录：**skill 本体**（随 skill 分发，通用）和**被测项目根的 `web-app-test/`**（项目专属实例）。

```
# ① skill 本体（通用，随 skill 分发，对所有项目只有一份）
<skill-dir>/
├── SKILL.md                          # 本文件：通用原则（依赖 playwright-cli skill）
└── templates/                        # 通用脚手架：为新项目从零创建规格书与脚本的模板
    ├── README.md                     # 模板用法、占位符约定、核心约定
    ├── TEST_SPECIFICATION.template.md
    ├── run_regression.template.py
    ├── playwright.config.template.js
    └── standard-flow.spec.template.js

# ② 被测项目根的 web-app-test/（项目专属，随项目走，进项目自己的 git）
<被测项目根>/web-app-test/
├── TEST_SPECIFICATION.md             # 项目专属测试用例规格书（主文档，由模板生成）
├── L4_OPERATION_GUIDE.md             # 分支文档示例：L4 浏览器动态接入操作手册
├── <其他分支文档>.md                 # 按需为某类测试新建的详细说明
└── scripts/                          # 可执行的测试脚本（全部在规格书附录登记并链接）
    ├── run_regression.py
    ├── playwright.config.js
    └── <flow>.spec.js
```

新接入一个项目时，①已随 skill 就位，只需按「新项目落地流程」从 ① 的 `templates/` 生成 ②。
